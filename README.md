# Notifcation Gateway: SMTP to Pushover & Smarthost Relay

This script runs a lightweight, reliable, and modular email server (SMTP) that intercepts incoming emails and dynamically routes them. It can convert alerts into push notifications using the [Pushover](https://pushover.net/) service, or seamlessly relay them to upstream SMTP Smarthosts (like Gmail, SendGrid, or corporate exchanges).

It is designed to be a bridge for systems, scripts, network hardware (like routers or NVRs), or legacy software that only know how to send alerts via unencrypted port 25 email. Instead of cluttering your inbox or struggling with modern TLS/Auth requirements on legacy hardware, this gateway catches those emails, secures them, and routes them instantly.

## Key Features

* **Dual Routing Engine:** Route emails to mobile devices via Pushover API, or forward them to upstream email servers via Smarthost relays.
* **Web UI Control Panel:** Features a sleek, hot-reloading FASH-stack web interface that manages your configurations securely.
* **No Lost Alerts:** If the internet goes down or an upstream API rate-limits you, the script saves pending notifications to your hard drive and retries them automatically using exponential backoff until they succeed.
* **Smart Routing:** Route alerts to different destinations depending on the `To:` address or the `From:` address of the email. You can use exact strings or powerful case-insensitive Regular Expressions.
* **Image Attachments:** The gateway intercepts image files attached to incoming emails and passes them directly to Pushover (handling API size limits automatically) or retains them for Smarthost forwarding.
* **Clean HTML:** Pushover's native HTML support is very messy. This script acts like a mini web browser, stripping out invisible formatting and fixing line breaks so your notifications look perfectly formatted.
* **Secure:** Supports strict SMTP Authentication (so random devices on your network can't abuse it) and STARTTLS encryption generation.
* **Hot-Reloading:** Update your routing rules, add new devices, or selectively restart specific listener endpoints without turning the server off.
* **Real-Time Queue Management:** The UI now features a dedicated Queue Manager tab. You can view delayed or failed messages in real-time, inspect the exact error returned by the upstream API or Smarthost, and see the next scheduled exponential backoff attempt.
* **Instant Retries:** Stuck messages can now be permanently deleted or forcefully retried instantly from the UI without waiting for the natural backoff timer to expire.
* **Custom EHLO Handshakes:** Smarthost aliases now support a configurable `advertised_hostname` override to bypass strict upstream spam filters requiring specific EHLO banner domains.
* **Non-Blocking Delivery Engine:** The background queue worker now utilizes interruptible event loops. This eliminates thread-locking during long backoff delays, allowing the daemon to instantly shut down or hot-reload without timing out the system service.
* **Enhanced UI Defaults:** When constructing new email routing matrices, the system now automatically defaults to using your secure Vault Aliases for App Tokens, streamlining configuration.
* **Silent Library Hardening:** Upstream deprecation warnings and strict binary TLS probe noise generated by `aiosmtpd` are now actively filtered from standard `INFO` logs, ensuring a clean operational view.

## Requirements and Installation

**Debian / Ubuntu:**
```bash
sudo apt-get update
sudo apt-get install python3 python3-requests python3-cryptography python3-aiosmtpd python3-passlib
# For the Web UI:
sudo apt-get install python3-fastapi python3-uvicorn python3-jinja2 python3-multipart
```

**Alpine Linux:**

```bash
apk update
apk add python3 py3-requests py3-cryptography py3-aiosmtpd py3-passlib
# For the Web UI:
apk add py3-fastapi py3-uvicorn py3-jinja2 py3-multipart
```

*Note: `passlib` is technically optional. It is only required if you want to use secure Linux crypt hashes for your local SMTP Client passwords instead of standard SHA256.*

## Configuration Files

The modular gateway relies on a split-configuration architecture to keep your routing rules explicitly separated from your highly sensitive API tokens and passwords.

The environment variables `GATEWAY_CONFIG`, `UI_CONFIG`, `VAULT_FILE`, and `VAULT_META_FILE` **must** point directly to file paths.

Below are complete, fully populated sample configurations to demonstrate a system routing alerts to Pushover while relaying specific administrative emails through a Gmail Smarthost.

### 1. `config.json` (Core Routing & Infrastructure)

This file dictates network binds, global fallbacks, and the address mapping matrices.

```json
{
  "smtp": {
    "default_route": "pushover",
    "disable_persistence": false,
    "queue_dir": "data/queue",
    "hostname": "gateway.local",
    "max_retry_backoff": 21600,
    "loglevel": "INFO",
    "listeners": [
      {
        "bind": "0.0.0.0:25",
        "starttls": false
      },
      {
        "bind": "0.0.0.0:587",
        "hostname": "secure.gateway.local",
        "starttls": true
      }
    ],
    "auth": {
      "local_admin": "plain_text_password_auto_hashed_on_save",
      "camera_nvr": "$5$rounds=5000$saltstring$hashedpassword..."
    }
  },
  "pushover": {
    "user": "GlobalUserAlias",
    "token": "GlobalAppAlias",
    "attachments": true,
    "force_plaintext": false
  },
  "smarthost": {
    "globals": {
      "alias": "gmail_relay",
      "force_plaintext": false,
      "disable_attachments": false
    },
    "aliases": {
      "gmail_relay": {
        "hostname": "smtp.gmail.com",
        "port": 587,
        "advertised_hostname": "gateway.local",
        "starttls": true,
        "disable_tls_validation": false,
        "auth": true,
        "username": "my_email@gmail.com",
        "disable_attachments": false,
        "force_plaintext": false
      }
    }
  },
  "routes": {
    "fido@mydomain.com": {
      "match": "to",
      "method": "pushover",
      "token": "FidoAppToken",
      "priority": 1
    },
    "rex@mydomain.com": {
      "match": "from",
      "method": "smarthost",
      "smarthost_alias": "gmail_relay"
    },
    "regex:^buster-(alert|warn)@local\\.lan$": {
      "match": "to",
      "method": "pushover",
      "token": "BusterAppToken",
      "priority": 2,
      "retry": 30,
      "expire": 3600,
      "sound": "siren"
    }
  }
}
```

### 2. `vault.json` (The Secret Store)

This file holds your actual passwords and API tokens, mapped to the aliases used in `config.json`. The Web UI automatically abstracts these into memory so your passwords are never written into the main config file.

```json
{
  "app": {
    "GlobalAppAlias": "azG918237abckd...",
    "FidoAppToken": "aK81237xbcz...",
    "BusterAppToken": "aB19283mnbcv..."
  },
  "user": {
    "GlobalUserAlias": "uQ192837poqwe..."
  },
  "smarthost": {
    "gmail_relay": "super_secret_app_password"
  }
}
```

### 3. `vault_meta.json` (Vault Timestamps)

Maintains the epoch timestamps of when secrets were last modified in the UI.

```json
{
  "app": {
    "GlobalAppAlias": 1720938475,
    "FidoAppToken": 1720938475,
    "BusterAppToken": 1720938475
  },
  "user": {
    "GlobalUserAlias": 1720938475
  },
  "smarthost": {
    "gmail_relay": 1720938475
  }
}
```

### 4. `ui_config.json` (Control Panel Presentation)

Adjusts how the Control Panel binds to the network and formats tables.

```json
{
  "port": 8443,
  "timezone": "UTC",
  "date_format": "YYYY-MM-DD HH:mm:ss",
  "relative_time": true,
  "expand_adv": false,
  "https": true,
  "tls_cert": "",
  "tls_key": "",
  "vault_sort": "name_asc",
  "smtp_sort": "name_asc",
  "smarthost_sort": "alias_asc",
  "ui_loglevel": "INFO"
}
```

## Understanding Routing Constraints

### The `routes` Object

| Variable | Description |
| --- | --- |
| `match` | When to trigger the alert: `to` (recipient), `from` (sender), or `both`. Default is `to`. |
| `method` | The backend relay processor: `pushover` or `smarthost`. |
| `regex:` | Prefix your email key with `regex:` to have the engine parse it as a case-insensitive regular expression instead of an exact string. |

#### Pushover Method Constraints

| Variable | Scope | Description |
| --- | --- | --- |
| `user` | Global / Route | The Pushover user or group key (or Vault Alias). |
| `token` | Global / Route | Your Pushover application token (or Vault Alias). |
| `force_plaintext` | Global / Route | Set to `true` to skip HTML rendering entirely and use the raw text payload. |
| `attachments` | Global / Route | Set to `false` to strip extracted image file attachments from the payload. |
| `device` | Global / Route | Send the alert to a specific device name instead of all devices. |
| `sound` | Global / Route | The name of a supported Pushover sound to override your default choice. |
| `tags` | Global / Route | A comma-separated string of arbitrary tags, used to categorize alerts. |
| `priority` | Global / Route | A number between `-2` (lowest) and `2` (emergency) to adjust alert urgency. |
| `ttl` | Global / Route | Time to Live. Seconds the message will stay on the device before automatic deletion. |
| `retry` | Global / Route | **(Required if `priority` is 2)**. Seconds between API retries. Must be `>= 30`. |
| `expire` | Global / Route | **(Required if `priority` is 2)**. Seconds before the API gives up. Must be `<= 10800`. |
| `url` | Global / Route | A supplementary URL to show alongside your message (Max: 512 characters). |

#### Smarthost Method Constraints

| Variable | Scope | Description |
| --- | --- | --- |
| `smarthost_alias` | Route | The exact name of the alias defined in the `smarthost.aliases` block. |
| `force_plaintext` | Global / Route | Set to `true` to dynamically rebuild the MIME envelope and strip HTML entirely. |
| `disable_attachments` | Global / Route | Set to `true` to dynamically rebuild the MIME envelope without file attachments. |

## Live Reloading (Signals)

If you modify the configuration files via the terminal instead of the UI, you can manually trigger hot reloads via standard OS signals.

| Signal | Command Example | Action |
| --- | --- | --- |
| `SIGUSR2` | `kill -SIGUSR2 <pid>` | Reloads configurations to apply new routing rules, tokens, or passwords without dropping network connections. |
| `SIGUSR1` | `kill -SIGUSR1 <pid>` | Re-evaluates all listener endpoints. If a specific listener's configuration (or disk certificate SHA256 hash) has changed, it selectively restarts that listener port. |
| `SIGTERM` | `kill -SIGTERM <pid>` | Gracefully shuts the server down, safely writing pending emails to disk before exiting. |
